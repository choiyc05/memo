*   OWASP Top 10 리스트에서 1위 자리를 유지하고 있으며, 테스트된 애플리케이션의 **100%**에서 어떤 형태로든 이 취약점이 발견되었습니다.
*   **정의:** Access control란 사용자가 할당된 권한을 벗어난 행동을 하지 못하도록 하는 것.

---

### 2. 주요 취약점 유형
*   **Deny by Default:** 특정 기능이나 사용자에게만 허용되어야 할 액세스가 모든 사용자에게 열려 있는 경우
*   **우회 및 변조:** **URL** 수정, **Metadata** 조작, 또는 **API** 요청 변조를 통해 **Access Control** 검사를 우회하는 행위.
*   **Force Browsing:** 권한이 필요한 페이지의 **URL**을 추측하여 직접 접속.
*   **IDOR (Insecure Direct Object References):** 고유 식별자를 조작하여 다른 사용자의 계정을 조회하거나 수정하는 행위
*   **Elevation of Privilege(권한 상승):** 로그인하지 않은 상태에서 사용자처럼 행동하거나, 일반 사용자가 관리자 권한을 획득하는 경우
*   **Metadata 조작:** JWT(JSON Web Token)와 같은 액세스 제어 토큰이나 쿠키를 재사용하거나 변조하여 권한을 높이는 행위
*   **CORS 오설정:** 신뢰하지 않는 출처(origin)로부터 API 접근을 허용하는 경우
*   **POST**, **PUT**, **DELETE** 등 데이터 변경 요청에 대한 **Access Control**이 누락된 **API**.

---

### 3. 방어 및 예방 대책
접근 제어는 공격자가 수정할 수 없는 **신뢰할 수 있는 서버 측 코드**나 API에서 구현될 때만 효과적입니다.

*   **기본 거부(Deny by Default):** 공공 자원을 제외하고는 모든 접근을 기본적으로 차단해야 합니다.
*   **메커니즘 재사용:** 접근 제어 로직을 메커니즘을 중앙 집중화하여 애플리케이션 전체에서 재사용하고, **CORS** 사용은 최소화.
*   **모델 기반 제어:** 사용자가 모든 레코드를 수정하는 것이 아니라, 해당 레코드의 소유권(ownership)을 확인하도록 강제해야 합니다.
*   **로그 및 모니터링:** 접근 제어 실패를 로그로 기록하고, 반복적인 실패가 발생할 경우 관리자에게 알림을 보내야 합니다.
*   **세션 관리:** 로그아웃 시 서버 측에서 세션 식별자를 무효화하고, JWT는 짧은 유효 기간을 가지도록 설정해야 합니다. 필요한 경우 **Refresh Tokens**와 **OAuth** 사용,
*   **서버 설정 최적화:** 웹 서버의 디렉터리 리스팅 기능을 비활성화하고, `.git` `.env` 등과 같은 메타데이터나 백업 파일이 웹 루트에 노출되지 않도록 해야 합니다.

### 4. 공격 시나리오 예시
*   **시나리오 #1: 파라미터 조작을 통한 IDOR (Insecure Direct Object References)**
    *   **공격 방식:** 애플리케이션이 `https://example.com/app/accountInfo?acct=12345`와 같은 방식으로 계정 정보를 불러올 때, 공격자는 브라우저 주소창이나 프록시 도구에서 `acct` 값을 `12346`으로 변경합니다.
    *   **상세 원인:** 서버가 요청된 `12346` 계좌가 현재 로그인한 사용자의 것인지 검증하지 않고 SQL 쿼리에 그대로 사용할 경우, 공격자는 타인의 민감한 금융 정보를 조회할 수 있게 됩니다.

*   **시나리오 #2: Force Browsing (강제 브라우징)**
    *   **공격 방식:** 공격자가 일반 사용자 권한으로 로그인한 상태에서 관리자 페이지의 URL을 추측하거나(예: `/admin`, `/config`), 검색 엔진이나 소스 코드 메타데이터를 통해 노출된 페이지 주소를 직접 입력하여 접속을 시도합니다.
    *   **상세 원인:** 인증되지 않은 사용자나 권한이 낮은 사용자가 관리자 페이지에 접근 가능하도록 서버 측에서 **Access Control** 검사가 누락된 경우 발생합니다.

*   **시나리오 #3: Front-end 전용 Access Control 우회**
    *   **공격 방식:** 웹 페이지에서 JavaScript를 사용하여 "관리자 버튼"을 숨기거나 특정 페이지로의 리다이렉트를 막더라도, 공격자는 이를 무시합니다. 대신 `curl`과 같은 명령줄 도구를 사용하여 `curl https://example.com/app/admin_getappInfo`와 같이 서버 API에 직접 요청을 보냅니다.
    *   **상세 원인:** 모든 보안 로직이 클라이언트 측에만 구현되어 있어, 서버가 API 요청을 수신할 때 권한 확인을 수행하지 않기 때문에 발생합니다.

*   **Metadata 및 JWT 조작 (Metadata Manipulation)**
    *   **예시:** 공격자가 자신의 권한 정보가 담긴 **JWT(JSON Web Token)**를 가로챈 뒤, 그 내부의 `role` 값을 `user`에서 `admin`으로 수정(Tampering)하여 서버에 다시 전송합니다. 만약 서버가 토큰의 서명을 제대로 검증하지 않는다면 공격자는 관리자 권한을 획득하게 됩니다(**Elevation of Privilege**).

*   **API의 HTTP Method별 Access Control 누락**
    *   **예시:** 특정 게시물을 읽는 `GET` 요청은 모든 사용자에게 허용되어 있지만, 게시물을 삭제하는 `DELETE` 요청에 대한 권한 검사가 누락된 경우입니다. 공격자는 본인이 작성하지 않은 게시물에 대해 `DELETE` API를 직접 호출하여 데이터를 파괴할 수 있습니다.

*   **Metadata 및 백업 파일 노출 (Information Exposure)**
    *   **예시:** 웹 서버 루트에 `.git` 폴더나 `config.php.bak` 같은 백업 파일이 방치된 경우입니다. 공격자는 `https://example.com/.git/config`와 같은 경로로 직접 접근하여 소스 코드의 구조나 DB 접속 정보 등 민감한 정보를 획득할 수 있습니다.

*   **SSRF (Server-Side Request Forgery)**
    *   **예시:** 애플리케이션이 사용자로부터 입력받은 URL로 서버가 대신 요청을 보내는 기능을 제공할 때 발생합니다. 공격자는 이를 악용하여 서버 내부 네트워크에 있는 관리자용 API(`http://localhost:8080/admin`)에 서버의 권한으로 접근하도록 강제할 수 있습니다.

### 1. 비밀번호 저장 (Password Hashing)
비밀번호는 복호화가 가능한 "암호화"가 아니라, 되돌릴 수 없는 **"해싱(Hashing)"**을 사용해야 합니다. 자료에서는 공격자가 GPU 등을 이용해 빠르게 크랙하는 것을 방지하기 위해 **적응형(Adaptive) 솔트 해시 함수** 사용을 강력히 권장합니다. 

* **권장 알고리즘:**
    * **Argon2** (가장 권장됨)
    * **scrypt**
    * **PBKDF2-HMAC-SHA-512**
    * **bcrypt** (레거시 시스템에서 사용 중일 경우 보안 전문가의 조언 필요)
* **주의사항:**
    * 단순한 MD5나 SHA1 같은 오래된 해시 함수는 절대 사용하지 마세요.
    * 반드시 **솔트(Salt)**를 추가하여 레인보우 테이블 공격을 방지해야 합니다.

---

### 2. 민감 데이터 암호화 (Data at Rest)
주민등록번호, 신용카드 번호, 건강 기록 등 복호화가 필요한 민감 정보는 **대칭키 암호화**를 사용합니다. 

* **권장 알고리즘:**
    * **AES-256** (Authenticated Encryption 모드 권장) 
* **권장 모드:**
    * **GCM (Galois/Counter Mode)** 또는 **CCM**과 같은 **인증된 암호화(Authenticated Encryption)** 방식을 사용하세요. 단순 암호화보다 무결성까지 보장되어 더 안전합니다. 
* **피해야 할 것:**
    * **ECB (Electronic Codebook)** 모드는 패턴이 드러나므로 절대 금지입니다.
    * **CBC (Cipher Block Chaining)** 모드는 패딩 오라클 공격 등에 취약할 수 있으므로 가급적 지양합니다. 

---

### 3. 핵심 보안 체크리스트 (구현 시 필수 참고)
알고리즘 선택만큼 중요한 것은 **관리 방식**입니다.

* **키 관리 (Key Management):** 암호화 키를 소스 코드에 하드코딩하거나 리포지토리에 올리지 마세요.하드웨어 보안 모듈(HSM)이나 클라우드 기반 키 관리 서비스(KMS)를 사용하는 것이 좋습니다. 
* **난수 생성 (Randomness):** 초기화 벡터(IV)나 키 생성 시에는 반드시 **CSPRNG**(암호학적으로 안전한 의사 난수 생성기)를 사용해야 하며, 예측 가능한 시드값을 사용해서는 안 됩니다. 
* **데이터 최소화:** 필요 없는 민감 데이터는 저장하지 말고 즉시 삭제하거나 토큰화(Tokenization)를 고려하세요. 

**요약하자면:** 비밀번호는 **Argon2**, 일반 민감 데이터는 **AES-256-GCM**을 사용하는 것이 현재 OWASP 가이드라인에 부합하는 가장 안전한 선택입니다.
